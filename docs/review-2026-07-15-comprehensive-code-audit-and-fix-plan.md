## 一、结论

当前版本的主流程已经具备较完整的功能，现有自动化测试也全部通过，但仍存在若干未被测试覆盖的关键边界问题。本次确认了 **5 项 P1、10 项 P2、4 组 P3 问题**。

最需要优先处理的是：

1. 整理会话没有按当前 B 站账号校验归属，切换账号后仍能读取、调整甚至尝试执行其他账号的会话。
2. 执行方案在程序中断后不具备幂等恢复能力，已经移动成功的资源会被再次执行，目标收藏夹也会重复创建。
3. 单次按组合键查询约 1000 个资源时会触发 SQLite 表达式深度上限，较大的收藏夹可能直接无法预览、微调或执行。
4. 服务默认监听局域网且没有应用层鉴权，同一局域网内能访问端口的设备可调用删除、移动、切换账号等写接口。
5. `GET /execute/stream` 会启动不可逆的执行任务，不符合 GET 只读语义，也放大了浏览器自动重连、预取或误访问带来的风险。

在修复 P1 前，不建议把服务端口暴露到不可信局域网，也不建议依赖“执行中断后重新打开程序继续执行”的行为。

### 2.2 最小复现确认

以下问题不是静态推测，已用临时数据库和模拟 B 站客户端复现：

| 场景 | 实际结果 | 结论 |
|---|---|---|
| 当前账号为 B，读取账号 A 的会话 | 返回的会话归属仍为 A，A 的会话也出现在可恢复列表 | 会话账号隔离缺失 |

| 同一会话重新采集，第二次源资源已变化 | 会话资源同时保留旧资源和新资源；同一跳过项记录 2 次 | 重采集不是当前源的快照 |
| 删除提交成功、删除确认接口抛异常 | 扫描状态永久停在 `removing`，扫描和条目均无错误信息 | 清理异常恢复不完整 |

现有测试通过只能说明已覆盖路径没有回归，不能否定上述未覆盖场景。

## 三、问题清单

### P1：必须优先修复

#### P1-1 整理会话缺少账号归属校验

**证据**

- `core/storage.py:611-616` 的 `load_session()` 仅按 `session_id` 查询。
- `core/storage.py:639-646` 的 `list_sessions_by_status()` 没有账号条件。
- `core/session.py:920-928` 返回或恢复所有账号的会话。
- `main.py:741-1005` 的会话读取、进度、取消、调整、执行、跳过项删除、失败重试、微调、版本切换和空源收藏夹删除接口均未统一校验会话所属账号。
- 相比之下，清理任务接口已经通过 `account_id` 做了归属校验，说明两套任务的安全边界不一致。

**影响**

- 切换到账号 B 后可以看到账号 A 的历史会话和资源标题。
- 如果继续执行，代码会拿账号 B 的 Cookie 操作账号 A 保存的源收藏夹 ID，可能在 B 下创建同名目标收藏夹，并产生混乱的失败记录。
- 任何知道或拿到 `sid` 的调用者都可访问该会话；结合局域网无鉴权问题，风险进一步放大。

**修复方案**

1. 在存储层新增 `load_session_for_account(session_id, account_id)`、`list_sessions_by_status(statuses, account_id)`。
2. 在服务层新增唯一入口 `require_owned_session(sid)`：先要求当前账号存在，再校验 `session.account_id == active_account_id`；不匹配统一返回 `SESSION_NOT_FOUND`，避免泄露其他账号存在性。
3. 所有 `/api/session/{sid}` 及其子路由在读取或启动任务前调用该入口。
4. `ClassifySession` 绑定 `account_id`，关键写操作开始前再次断言 `bili.account_id` 与会话账号一致，防止路由层遗漏。
5. 可恢复会话和启动恢复逻辑只处理当前账号；账号切换后重新加载该账号的可恢复任务。

**验收**

- 建立账号 A、B 和各自会话，逐一调用全部会话接口；B 访问 A 的会话均返回 404/业务等价错误且数据库、B 站模拟调用均无变化。
- 覆盖普通请求、SSE、后台任务复用、版本激活和删除接口。

#### P1-2 执行中断后会重复创建收藏夹并重放已成功移动的资源

**证据**

- `core/session.py:571-613` 每次执行都从完整方案和完整 `session_video_sources` 重新构建移动分组，没有过滤 `moved=1` 或 `executed=1`。
- `core/session.py:614-648` 每次执行都重新创建分类对应的目标收藏夹，目标 `fid` 没有持久化。
- `core/session.py:923-928` 启动恢复只是把 `executing` 改回 `pending_review`。
- 已通过最小复现确认：成功状态已经写入后再次执行，仍会创建目标收藏夹并调用移动接口。

**影响**

- 程序在“B 站移动成功、数据库写入前”或部分批次完成后崩溃，重启后无法判断真实进度。
- 用户重试可能得到重复的目标收藏夹、重复失败项和不可信的成功/失败统计。
- 多源收藏夹下，同一资源的不同源位置需要独立确认，单个方案项的 `executed` 无法完整表示位置级进度。

**修复方案**

1. 新增 `execution_runs`：记录 `run_id/session_id/version_id/account_id/status/processed/total/error/started_at/finished_at`。
2. 新增 `execution_targets`：以 `(run_id, category)` 唯一，持久化 `target_fid/create_status/create_error`；恢复时复用目标收藏夹，不再重新创建。
3. 把 `(session_id, resource_id, resource_type, source_fid)` 作为最小执行单元，只构建 `moved=0` 的待执行分组。
4. 每批移动后立即按源位置写入成功；计划项只有在所有源位置均成功后才标记 `executed=1`。
5. 启动恢复时执行远端对账：检查资源是否仍在源收藏夹、是否已在持久化目标收藏夹；据此补写成功或保留待重试状态。
6. 失败项增加位置唯一约束或执行批次标识，避免每次恢复重复插入相同错误。
7. 状态改为 `executing -> execution_interrupted -> pending_review/done`，不要直接无条件回退。

**验收**

- 对以下崩溃点分别测试恢复：目标收藏夹创建后、B 站移动成功前、B 站移动成功后但本地落库前、本地成功落库后。
- 每个分类最多创建一个目标收藏夹；每个源位置最多确认移动一次；统计与远端最终状态一致。

#### P1-3 约 1000 个资源时 SQLite 查询会直接失败

**证据**

- `core/storage.py:583-599` 为每个资源键生成一个 `(resource_id = ? AND resource_type = ?)`，再用 `OR` 连接。
- 该方法被分类、预览、微调、重试和执行等完整资源集合调用。
- 1001 个组合键最小复现得到：`Expression tree is too large (maximum depth 1000)`。

**影响**

- 选择多个收藏夹汇总超过约 1000 条时，可能在采集完成后、预览或执行阶段突然失败。
- README 当前只提示大收藏夹可能较慢，没有说明会触发硬错误。

**修复方案**

1. 先对资源键去重。
2. 优先使用 SQLite 行值查询：`(resource_id, resource_type) IN ((?, ?), ...)`。
3. 同时按 300～400 个键分块，避免变量数量和不同 SQLite 构建参数的上限；各块结果合并后按资源键稳定排序。
4. 如果后续集合规模继续增长，可使用临时表并 `JOIN`，避免动态超长 SQL。
5. 所有接收大量 ID 的其他查询也使用统一分块工具，尤其是清理条目、跳过条目和批量资源查询。

**验收**

- 0、1、999、1000、2000、5000 个资源键均可查询。
- 同一 ID 不同 `resource_type` 不串数据，重复输入不产生重复结果，账号条件始终生效。

#### P1-4 默认局域网暴露全部写接口且没有应用鉴权

**证据**

- `main.py:1018` 默认监听 `0.0.0.0:8765`。
- `README.md:72` 明确说明局域网设备可以访问。
- 应用没有登录口令、配对令牌、会话 Cookie、Origin/Host 校验或 CSRF 防护。
- API 可以读取收藏内容，并执行移动资源、删除失效项、删除收藏夹、切换账号和退出账号等操作。

**影响**

- 在办公网络、公共 Wi-Fi、虚拟网卡或端口被错误转发时，其他设备可能直接控制当前 B 站账号收藏。
- 即便浏览器界面有二次确认，调用者也可直接请求后端接口绕过前端确认。

**修复方案**

1. 默认监听改为 `127.0.0.1`。
2. 设置中增加明确的“允许局域网访问”开关；启用时生成随机配对口令/访问令牌，不允许无令牌请求。
3. 使用 `HttpOnly + SameSite=Strict` 的本地会话 Cookie，或每次请求携带短期 Bearer Token；所有 API 和 SSE 都校验。
4. 校验允许的 `Host` 和 `Origin`，写操作增加 CSRF 防护。
5. 启动页只显示一次局域网访问地址和配对码；提供立即吊销入口。
6. README 和 BAT 文案同步说明默认仅本机访问。

**验收**

- 默认情况下其他局域网设备无法连接。
- 开启局域网模式后，无令牌或错误 Origin 的普通请求、SSE 和写请求均返回 401/403。

#### P1-5 GET 进度流会启动不可逆执行

**证据**

- `main.py:819-877` 的 `GET /api/session/{sid}/execute/stream` 在状态为 `pending_review` 时调用 `_get_or_start_execution()`。
- `static/app.js:1860-1873` 直接建立该 EventSource，实际执行由 GET 触发。
- 虽然另有 `POST /api/session/{sid}/execute`，但前端没有采用“POST 启动、GET 观察”的任务协议。

**影响**

- 浏览器自动重连、预取、代理重试或用户直接访问 URL 都可能触发移动和创建收藏夹。
- GET 不应产生副作用，否则审计、幂等和访问控制都更难保证。

**修复方案**

1. `POST /api/session/{sid}/execute` 只负责启动，返回 `202 + job_id + reused`，并接收一次性确认令牌或幂等键。
2. `GET /api/session/{sid}/execute/stream?job_id=...` 只观察已有任务，不存在时返回错误，绝不创建任务。
3. 新增 `GET /api/session/{sid}/execute/active` 用于刷新或重新进入页面后恢复任务。
4. 前端二次确认成功后先 POST，再连接 SSE。

**验收**

- 单独 GET 进度流不会创建收藏夹、移动资源或改变会话状态。
- 重复 POST 使用同一幂等键只复用同一任务。

### P2：重要可靠性与体验问题

#### P2-1 重采集会混入旧资源并重复追加跳过项

**证据**

- `core/session.py:93-227` 采集开始前不清理或隔离上一次采集产物。
- `core/storage.py:848-854` 对会话资源使用 `INSERT OR IGNORE`，旧资源不会删除。
- `core/storage.py:902-910` 跳过项每次直接插入，没有尝试编号或唯一约束。
- `core/session.py:923-928` 把中断的采集/分类会话退回 `draft`，随后会重新全量采集。
- 最小复现中第一次的资源 1、第二次的资源 3 同时留存，同一跳过项出现两条。

**修复方案**

1. 增加 `collection_generation` 或 `collection_runs`，采集过程写入新代次。
2. 只有全部源采集成功后，才原子切换会话的激活代次；失败或取消时旧快照仍可用。
3. 切换成功后清理旧代次的 `session_video_sources`、`skipped_items` 和中间分类产物。
4. 跳过项唯一键至少包含 `(session_id, generation, source_fid, resource_id, resource_type, reason_code)`。
5. 如果不采用代次表，至少在明确的“从头重试”事务中清空全部派生数据，不能仅改状态。

#### P2-2 切换账号、退出账号和新增账号未统一约束后台任务

**证据**

- `main.py:423-428` 切换账号只检查采集/分类和执行任务，没有检查微调、清理任务。
- `main.py:404-415` 退出账号不检查任何后台任务。
- `main.py:75-104` 新账号扫码成功后直接激活新账号，同样绕过任务检查。
- 已创建的 Manager 和 BilibiliClient 会继续持有旧账号 Cookie，因此界面账号与后台任务实际账号可能短时不一致。

**修复方案**

1. 建立统一 `TaskRegistry`，每个任务记录 `kind/account_id/session_id/scan_id/status/task`。
2. 切换、退出和新账号激活都走同一 `activate_account_safely()`。
3. 默认策略：当前账号存在任何采集、分类、执行、微调、扫描或删除任务时禁止切换/退出；如未来允许取消，应先取消并 `await gather` 完成收尾。
4. 后台任务启动时固定账号上下文，不再依赖任务运行期间变化的“当前账号”。

#### P2-3 清理删除异常或取消后状态会卡在 `removing`

**证据**

- `core/cleanup.py:155-223` 的 `remove()` 在开始时写入 `removing`，但没有像 `scan()` 那样捕获 `CancelledError` 和普通异常。
- 删除提交后的远端确认一旦抛异常，函数直接退出，扫描和条目均不记录失败原因。
- 已用模拟客户端复现：确认接口异常后状态仍为 `removing`，错误字段为空。

**修复方案**

1. 用 `try/except asyncio.CancelledError/except Exception` 包住整个删除流程。
2. 取消时写 `cancelled`；异常时写 `failed + error`，并给已提交但未确认的条目写入“状态未知，需重新核验”。
3. 重新删除前先读取远端 ID，已不在源收藏夹的条目直接补记成功，避免重复提交。
4. `failed/cancelled` 状态允许仅重试未成功条目。

#### P2-4 后台任务和扫码登录对象会长期滞留内存

**证据**

- `main.py:58-61` 使用全局字典保存任务。
- `main.py:194-203` 只移除 `_running_refinements`，不移除 `_refinement_jobs`；任务还持有完整方案结果。
- `main.py:207-245` 的 `_cleanup_jobs` 完成后也不清理。
- `main.py:452-470` 只有扫码成功才删除 `_pending_logins`；二维码过期、用户离开页面或程序长期运行时，临时客户端和临时 Cookie 文件会残留。

**修复方案**

1. 终态任务只保留轻量快照，按 10～30 分钟 TTL 或固定数量清理。
2. 完整方案从数据库重新读取，不保存在任务字典中。
3. 登录会话增加过期时间和取消接口；过期、取消、失败和启动清理时均删除临时文件。
4. FastAPI lifespan 退出阶段取消并等待全部任务，关闭所有 HTTP 客户端。

#### P2-5 离开预览页后微调进度不会自动恢复

**证据**

- `static/app.js:75-89` 切换任何页面都会关闭唯一的 `activeEventSource`。
- `static/app.js:1620-1682` SSE 建连逻辑只存在于点击“生成新方案”的函数内部。
- `static/app.js:1905-1934` 返回预览页后只显示最后一次内存快照，没有重新连接任务；刷新页面后 `job_id` 也会丢失。

**修复方案**

1. 抽出 `connectRefineStream(sid, jobId)`，允许幂等重连。
2. 增加 `GET /api/session/{sid}/refine/active`，返回当前账号、当前会话的运行任务和最近进度。
3. 返回预览页或页面刷新时主动查询并重连；完成后从数据库加载新激活版本。
4. 不再用单一 EventSource 管理所有业务流，至少按 `pipeline/execution/refine/cleanup` 分槽管理。

#### P2-6 非 AI 操作被不必要地绑定到完整 AI 配置

**证据**

- `main.py:267-275` 的 `get_session_mgr()` 总会调用 `get_ai()`。
- 因此查看现有方案、调整分类、执行、查看失败项等不需要 AI 的接口，也会在 AI 配置缺失时失败。
- `main.py:289-294` 只有 AI 配置完整时才执行启动恢复，执行中断恢复也被跳过。

**修复方案**

1. `ClassifySession` 允许 `ai=None`，仅 `classify/refine/retry_unclassified` 在入口调用 `require_ai()`。
2. 将 Manager 构造拆为只需 B 站客户端的基础依赖，以及需要 AI 的分类依赖。
3. lifespan 始终执行会话和任务状态恢复；只有真正需要继续 AI 分类时才要求配置。

#### P2-7 B 站 HTTP 客户端未复用，完整模式逐条串行补充信息

**证据**

- `core/bilibili_api.py:74-75` 每次请求新建 `httpx.AsyncClient`。
- `main.py:64-72` 每次获取当前账号客户端时又新建 `BilibiliClient`。
- `core/session.py:146-153` 完整模式逐条串行调用补充信息。
- `core/bilibili_api.py:475-492` 每个视频需要查看信息和标签两次请求，多源重复资源也没有先统一去重。

**影响**

- 连接池、DNS 和 TLS 无法复用；完整模式资源越多，等待时间近似线性增加。
- 频繁创建客户端会增加连接资源和限流压力。

**修复方案**

1. 建立账号级长生命周期 `BilibiliClient` 和单个 `httpx.AsyncClient`，在 lifespan 结束时关闭。
2. 补充信息前按资源组合键去重并读取缓存。
3. 使用 `Semaphore(4～6)` 做有限并发，增加指数退避和账号级限速器，不能无上限并发请求 B 站。

#### P2-8 前端运行依赖两个外部 CDN

**证据**

- `static/index.html:365-366` 运行时从 jsDelivr 加载 Tailwind Browser，并从 unpkg 加载 Lucide。

**影响**

- 网络不稳定、DNS/CDN 被阻断或离线时，界面可能无样式、无图标或启动明显变慢。
- Tailwind Browser 在用户浏览器中运行时编译，不适合作为稳定的桌面工具生产依赖。

**修复方案**

1. 构建并提交固定版本的本地 CSS；只打包实际使用的 Lucide 图标。
2. 移除运行时 Tailwind 编译器和外部脚本硬依赖。
3. 增加 CSP；若保留 CDN 兜底，应固定版本并使用 SRI。

#### P2-9 排序和收藏夹删除只立即确认一次，可能误报失败

**证据**

- `main.py:515-535` 排序后立即读取一次并要求顺序完全一致。
- `main.py:539-596` 单个和批量删除后也立即读取一次。
- 清理资源删除已经使用 `0.5/1/2` 秒重试，说明项目本身已考虑 B 站最终一致性，但收藏夹接口没有复用该策略。

**修复方案**

1. 抽出统一远端确认器，按 `0.5/1/2` 秒有限重试。
2. 接口区分“提交失败”和“提交成功但尚未确认”；后者前端提示稍后自动刷新，不应直接当作操作失败。

#### P2-10 采集/分类失败没有持久化错误状态

**证据**

- `core/session.py:82-87` 的流水线没有统一异常收尾。
- `main.py:787-791` 只通过当前 SSE 返回异常；断线或刷新后错误信息丢失。
- 状态机没有 `failed` 或 `retryable_failed`，启动时只能把中间状态退回 `draft`，进而触发 P2-1 的全量重采集问题。

**修复方案**

1. 给会话增加 `failed_stage/error_code/error_message/attempt_id`，或新增 `session_attempts`。
2. 流水线顶层持久化 `failed/cancelled` 终态。
3. 提供显式重试接口，并根据失败阶段决定从采集快照、AI 批次还是方案保存继续。

### P3：健壮性、迁移与维护问题

#### P3-1 API 输入约束不足，且 `default_privacy` 是无效配置

**证据**

- `main.py:308-345` 的 `mode`、分类名称、微调指令和 ID 列表没有严格枚举、去空白、长度或数量限制。
- `main.py:304/482` 保存并返回 `default_privacy`，但 `core/session.py:647/811/883` 创建收藏夹始终硬编码 `privacy=1`。

**修复方案**

- `mode` 改为 `Literal['quick', 'full']`；文本增加 `strip + min/max length`；ID 列表去重并限制最大数量。
- 分类名称按 B 站实际限制统一校验，禁止空名称和仅空白名称。
- 要么完整接入 `default_privacy`，要么从配置模型、文件和 README 删除，避免虚假设置。

#### P3-2 数据库迁移对半迁移状态和备份碰撞的防御不完整

**证据**

- `core/storage.py:320-364` 的资源键迁移在已有 `resource_id` 但没有 `account_id` 的异常旧表上仍直接读取 `account_id`。
- `core/storage.py:478-518` 的 classifications 半迁移表若只有 `resource_id`、没有 `resource_type`，会回退为 `avid`，可能丢失已保存的非视频资源 ID。
- `core/storage.py:521-528` 的备份名精确到秒，同一秒内多次破坏性迁移会覆盖同名备份；当前仓库已有相邻秒备份，说明一次启动可能执行多次重建。

**修复方案**

1. 使用通用迁移读取器，根据每个实际存在列逐项保留，不假定固定历史形态。
2. 备份文件名加入微秒或 UUID，并使用不覆盖创建模式。
3. 一次启动的多表迁移只做一份迁移前完整备份，并在事务中完成全部表重建。
4. 建立旧版、半迁移、重复启动和故障回滚的迁移矩阵测试。

#### P3-3 SQLite 查询和事件循环负载仍可优化

**证据**

- `core/cleanup.py:71/111` 每页通过读取全部清理项重新计算问题总数。
- `skipped_items(session_id)`、`failed_items(session_id)`、方案版本等常用外键查询缺少显式索引。
- 同步 SQLite 操作直接运行在 FastAPI 事件循环线程；大批量逐条写入时会阻塞 SSE 心跳和其他请求。

**修复方案**

1. 扫描时增量维护问题数，或使用 `COUNT(*)`。
2. 为常用会话、扫描和版本过滤列增加索引，并用 `EXPLAIN QUERY PLAN` 验证。
3. 批量写入放入单个事务；规模较大时通过单独数据库执行器线程或异步 SQLite 封装避免阻塞事件循环。

#### P3-4 创建失败会遗留草稿，分页缺少“无进展”保护

**证据**

- `core/session.py:58-68` 先创建会话，再请求并校验源收藏夹；B 站接口失败或源收藏夹不属于当前账号时会留下无来源的草稿会话。
- `core/bilibili_api.py:227-304/317-406` 主要依赖 `has_more` 和空页终止；如果上游异常地重复返回同一非空页且 `has_more=true`，会一直请求。

**修复方案**

1. 先校验并取得收藏夹快照，再在事务中创建会话和来源；失败时不留下半成品。
2. 分页记录页签名或资源键集合；连续无新增资源时中止并记录上游分页异常。
3. 根据已声明总数设置合理最大页数并留冗余，超限时返回可诊断错误。
