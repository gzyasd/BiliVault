## 二、发现的问题

### 问题 P1：AI 接口异常未翻译成前端可读错误（中等）

**位置**：`core/ai_classifier.py:45-55`、`main.py:65-67`

**现象**：
`_chat_json` 只捕获 `json.JSONDecodeError` 并抛 `AiApiError`。但 OpenAI SDK 的其他异常（如 401 鉴权失败 `AuthenticationError`、429 限流 `RateLimitError`、超时 `APITimeoutError`、连接错误 `APIConnectionError`）**不是 `AiApiError` 子类**，会直接冒泡到 FastAPI。

`main.py` 的全局异常处理器只处理 `BibiError` 子类：
```python
@app.exception_handler(BibiError)
async def bibi_error_handler(_, exc: BibiError):
    return JSONResponse({"code": exc.code, "message": exc.user_message}, status_code=400)
```

OpenAI SDK 的异常不是 `BibiError` 子类，错误表现取决于调用路径：

- 若通过普通 HTTP 路由触发，会变成 HTTP 500 或默认错误响应，前端通常只能看到类似 `请求失败 500` 的泛化提示。
- 当前主流程主要通过 SSE 的 `/api/session/{sid}/stream` 触发 AI 分类；该路由会捕获 task exception 并发送 `fail` 事件，但消息仍是未结构化的底层异常文本，不会稳定映射为"API Key 无效"、"网络问题"或"限流"。

因此本问题的核心不是“一定 HTTP 500”，而是 **AI 错误没有被统一翻译成前端可读、可处理的业务错误码**。

**与设计文档的偏差**：设计文档第8节明确要求：
- AI 接口 401 → 前端提示"API Key 无效"，跳设置页
- AI 接口 429 → 指数退避重试 3 次

**建议修复**：
1. 在 `ai_classifier.py` 的 `_chat_json` 外层加 try/except，捕获 `openai.AuthenticationError`/`openai.RateLimitError`/`openai.APITimeoutError`/`openai.APIConnectionError`，翻译成带特定 code 的 `AiApiError`（如 `AI_AUTH_FAILED`/`AI_RATE_LIMIT`/`AI_TIMEOUT`）
2. 或在 `main.py` 增加这些 OpenAI 异常的 exception_handler；但更推荐在 `ai_classifier.py` 内部翻译，因为 SSE 与普通路由都能复用同一套错误语义。

### 问题 P2：AI 调用缺少重试机制（中等）

**位置**：`core/ai_classifier.py:45-55`

**现象**：
`_chat_json` 调用 `self._client().chat.completions.create(...)` 没有任何重试。设计文档第6.3节要求：
- AI 调用超时/限流：自动重试 3 次，指数退避
- AI 接口 429 限流：指数退避重试 3 次

实际使用时，DeepSeek/通义等偶发限流可能直接导致分类流程失败；如果异常被转换成 `AiApiError` 并在 `classify_batch` 中兜底，也可能导致该批视频全部标"未分类"。无论哪种表现，缺少重试都会放大临时限流/超时带来的失败率。

**建议修复**：
在 `_chat_json` 包一层重试循环，捕获限流/超时类异常，sleep `1s * 2^attempt` 后重试，最多 3 次。

### 问题 P3：.gitignore 缺少 .libs 目录（中等）

**位置**：`.gitignore`

**现象**：
项目根目录有 `.libs/` 目录（约 100+ MB，包含 PIL/pytest/fastapi/openai 等所有依赖的 vendor 源码，是审查前的子代理因沙箱写入限制创建的）。当前 `.gitignore` 排除了 `.venv/` 但**没有排除 `.libs/`**：

```
config.json
bilibili_cookie.json
bibi.db
__pycache__/
.venv/
*.pyc
.pytest_cache/
```

如果 `git add .` 会把整个 vendor 依赖树提交，污染仓库。

**建议修复**：在 `.gitignore` 增加 `.libs/`。同时建议 README 说明 `.libs/` 是环境受限时的备选依赖目录，正常用户用 `pip install -r requirements.txt` 即可，无需 `.libs/`。

### 问题 P4：NOT_LOGGED_IN 前端处理不统一（低）

**位置**：`static/app.js`

**现象**：
只有 `renderHome`（第135-141行）处理了 `NOT_LOGGED_IN` 错误码并跳登录页。其他可能触发该错误的调用没有处理：
- `newSession`（第157行）调 `/api/session` 时 cookie 可能刚好过期
- `openSession`（第164行）调 `/api/session/{sid}`
- `execute`（第302-312行）调 `/api/session/{sid}/execute`
- `adjustItem`（第315行）调 `/api/session/{sid}/adjust`

这些地方失效时只会 `alert(e.message)`，用户看到"B站登录已失效"但不会自动跳登录页，需要手动点导航栏设置→返回。

**建议修复**：
在 `api()` 函数里统一处理：捕获 `NOT_LOGGED_IN` code 时自动 `showView('login'); renderLogin();` 并 rethrow 中断调用链。这样所有调用点都受益。

### 问题 P5：SQLite 无 busy_timeout，并发可能锁冲突（低）

**位置**：`core/storage.py:106-109`

**现象**：
`_conn()` 创建连接时没有设置 `PRAGMA busy_timeout`。SQLite 默认 busy_timeout 为 0，遇到锁立即抛 `database is locked`。

虽然这是单用户本地工具，但 FastAPI 异步 + SSE 场景下存在并发：SSE 流端点在后台跑 `run_pipeline`（写 videos/classifications），同时用户可能在预览页调 `/api/session/{sid}/adjust`（写 classifications）。两个请求用不同的 sqlite3 连接，可能偶发锁冲突。

**建议修复**：
```python
def _conn(self) -> sqlite3.Connection:
    conn = sqlite3.connect(self._db_path)
    conn.row_factory = sqlite3.Row
    conn.execute("PRAGMA busy_timeout = 5000")
    return conn
```

### 问题 P6：SSE 流客户端断开后 task 继续运行，但缺少同 session 去重（低，建议按推荐方案修）

**位置**：`main.py:136-162`

**现象**：
`api_session_stream` 用 `asyncio.create_task(mgr.run_pipeline(...))` 启动后台任务。如果用户关闭浏览器，EventSource 断开，`event_gen()` 生成器收到 `GeneratorExit`，但 `task` 不会被自动取消——`run_pipeline` 会继续在后台跑完（包括调 AI、写 DB）。

**正面影响**：这其实是断点续作的隐性保障——即使用户关浏览器，分类仍在后台完成，下次打开能从 `pending_review` 继续。

**负面影响**：task 引用丢失但仍在运行，如果用户反复开关浏览器或同一个进度页被多次打开，可能为同一个 `session_id` 启动多个 `run_pipeline`，造成重复拉取、重复 AI 调用、重复写 DB 或状态冲突。

**推荐做法**：不要在 SSE 断开时直接取消后台任务。更符合本工具场景的方案是：**断开后继续跑，但加 session 级任务注册表，防止同一个 session 重复启动多个 pipeline**。

建议实现要点：

1. 在 `main.py` 中维护进程内任务表，例如 `running_sessions: dict[str, asyncio.Task]`。
2. `/api/session/{sid}/stream` 被打开时：
   - 如果 `sid` 已有未完成 task，则复用该 task，不再新建 `run_pipeline`。
   - 如果没有 task，且 session 状态仍是 `draft/collecting/classifying`，才创建 task 并登记。
   - 如果 session 已经是 `pending_review/done`，不要再启动 pipeline，直接发送 `done` 或当前状态。
3. task 完成后从 `running_sessions` 删除。
4. 如果要支持多浏览器同时看进度，需要把进度事件从单 queue 扩展为每连接一个 queue；若暂不做广播，也至少应在重连时从 DB 返回当前 session 状态，避免用户误以为任务卡住。

不推荐的做法：

- 不建议在 `event_gen finally` 中无条件 `task.cancel()`。这会让浏览器刷新、手机锁屏、网络闪断都中断整理，削弱断点续作价值。
- 不建议继续保留当前“每次 SSE 连接都 create_task”的方式；它是重复任务的根源。

### 问题 P7：状态机 "failed" 状态是死代码（低，文档一致性问题）

**位置**：`core/session.py:8-16`、`session.py:167`

**现象**：
`_VALID_TRANSITIONS` 定义了 `"failed": {"pending_review"}` 状态，但 `execute` 方法无论失败多少项，最后都转到 `"done"`：
```python
self.storage.update_session_status(sid, "done", stats=stats)
```
永远不会转到 `"failed"`。`list_resumable` 包含 `"failed"`，`retry_failed` 要求 `status == "done"` 才能重试——如果将来有人手动把状态改成 `failed`，retry 会失败。

**建议**：要么在 execute 中当 `failed > 0` 时转 `failed` 状态（然后 retry_failed 放宽到接受 `failed`），要么从 `_VALID_TRANSITIONS` 和 `list_resumable` 移除 `failed`。保持状态机与实际行为一致。

---

### 问题 P8：collect/classify 阶段异常后会话状态卡住（低）

**位置**：`core/session.py:39-54`、`main.py:155-158`

**现象**：
`collect` 先 `_transition(sid, ..., "collecting")`，然后调 `bili.get_folder_videos`。如果中途 B 站返回 -403 风控或网络断开，异常冒泡，会话状态停在 `"collecting"`。同理 `classify` 阶段异常会停在 `"classifying"`。

`resume_on_startup` 只在**重启时**才回退这些状态。当前会话卡住后，前端 `fail` 事件后回首页，用户再选收藏夹会新建会话，旧会话卡在 `collecting` 直到重启。

**建议（可选）**：在 `run_pipeline` 外层 try/except，捕获异常时把状态回退到 `draft`。或在前端 `fail` handler 里调一个回退 API。对一次性工具，重启清理也可接受。

### 整体评价

项目实现质量良好，审查团队的改进覆盖了 WBI 签名健壮性、扫码状态兼容、完整模式、错误处理、XSS 防护、测试覆盖等关键点。二次复核时当前测试结果为 `47 passed`。

剩余问题主要集中在**AI 错误处理链路**（P1/P2）——这是实际使用时最先暴露的痛点，建议优先修复 P1 和 P2。P3 是仓库卫生问题，应尽快修复避免误提交大量 vendor 代码。P4/P5 是体验和健壮性优化。P6 不建议简单取消 task，推荐用 session 级任务去重保留断点续作体验。P7/P8 影响有限，可酌情处理。

### 建议修复顺序

1. **P3**（.gitignore 加 .libs）——1 行改动，立即修复
2. **P1**（AI 异常翻译）——在 `ai_classifier.py` 捕获 OpenAI 异常转 `AiApiError` 带特定 code
3. **P2**（AI 重试）——在 `_chat_json` 加退避重试
4. **P4**（前端统一 NOT_LOGGED_IN）——在 `api()` 统一处理
5. **P5**（SQLite busy_timeout）——1 行改动
6. **P6**（SSE 任务注册表）——保留断开后继续跑，但禁止同一 session 重复启动 pipeline
7. P7（状态机清理）——可选
