# 无访问权限收藏条目处理设计

## 背景

用户选择的五个源收藏夹在收藏夹列表接口中显示共 12 个条目，但资源详情列表接口返回空数组。现网核验表明：资源 ID 接口仍返回 12 个 `id:type` 和 BVID；逐条查询视频详情均返回 B 站错误码 `62012`（仅 UP 主自己可见）。

现有程序在本次会话没有采集到资源时，会按第一个源收藏夹读取历史 `videos` 缓存。该缓存查询没有按账号限定，导致本次 12 条源条目错误显示为 675 个可整理条目，并可能将无关历史数据提交给 AI。

## 目标

1. 本次会话的可整理数量只能来自本次采集到的会话资源，绝不使用历史缓存补位。
2. 对“仍存在收藏关联、但当前账号不能读取详情”的资源，识别为跳过条目并保留来源收藏夹、资源 ID、类型和 BVID。
3. 在预览页按原因展示此类条目，标题缺失时显示 `无法访问的视频（BVID：...）`，支持勾选并二次确认后从原收藏夹移除。
4. 若源收藏夹没有可整理资源，仍进入预览页展示跳过条目；不调用 AI，不创建任何目标收藏夹。

## 非目标

- 不尝试绕过 B 站的作品可见性限制。
- 不对每条无法访问资源逐个请求详情接口以区分 `62012`、审核中或已删除。资源详情缺失统一标记为“无访问权限或无法读取”，避免大量额外请求和限流风险。
- 不自动删除任何收藏条目。

## 方案选择

采用“资源 ID 清单与详情清单做差”的方案。

每个源收藏夹均调用两类 B 站接口：

- `/x/v3/fav/resource/list`：取得当前账号可读取详情的资源，用于正常分类。
- `/x/v3/fav/resource/ids`：取得收藏关联的完整 `id:type` 清单及可用 BVID。

以 `(resource_id, resource_type)` 作为组合键。ID 清单存在、详情清单缺失的键即为无法读取条目。该方案每个源收藏夹只新增一次请求，能够覆盖“全部隐藏”和“可见、不可见混合”两种情况。

## 后端设计

### BilibiliClient

新增 `get_folder_resource_ids(fid, storage=None)`：

- 调用 `/x/v3/fav/resource/ids`。
- 标准化返回为 `{resource_id, resource_type, bvid}` 的列表。
- 忽略没有资源 ID 的异常响应项。

### 采集流程

`ClassifySession.collect()` 对每个源收藏夹维护：

- `visible_keys`：详情列表中已成功读取的资源组合键。
- `source_raw_count`：详情列表实际扫描数量。
- `source_collected`：可送入 AI 的条目数量。
- `source_skipped`：失效、无 ID、无访问权限等跳过数量。

详情列表分页结束后，调用资源 ID 清单接口并执行以下逻辑：

1. 计算 `missing_keys = id_keys - visible_keys`。
2. 为每个 `missing_key` 写入 `skipped_items`：
   - `reason_code = inaccessible`；
   - `reason_label = 无访问权限`；
   - `detail = B站未返回资源详情，可能仅UP主可见或受权限限制`；
   - `removable = True`；
   - 视频条目的 `avid` 保存资源 ID；非视频条目也沿用现有 `avid` 字段保存资源 ID，并以 `resource_type` 消除歧义。
3. 将源收藏夹已扫描数校正为 `max(source_raw_count, len(id_keys))`，使进度与“收藏条目”一致。
4. 更新该源及会话的收集、跳过统计，并再发送一次采集进度事件。

若资源 ID 清单接口失败：保留已经读取到的正常详情条目，不生成无法删除的猜测性条目；记录该源未完成交叉核验的警告日志。无论该接口是否成功，都不得回退到历史视频缓存。

### 分类与预览

- `classify()` 仅通过 `session_video_sources` 的 `(resource_id, resource_type)` 查询本会话缓存。
- 当会话资源为空时：跳过 `ai.classify()` 调用，保存空分类结果和空方案版本，进入 `pending_review`。
- `get_plan()` 也不再按 `source_fid` 读取 `videos` 历史缓存。
- 因此预览页的“可整理”始终等于当前会话可读取资源数；无法访问资源只出现在跳过面板。

### 删除

现有 `remove_skipped_items()` 和 `/api/session/{sid}/skipped-items/remove` 已满足接口形态，无需另建删除接口：

- 仅删除 `removable = True`、尚未删除且有来源收藏夹和资源 ID 的条目。
- 按 `source_fid` 分组、每批最多 50 个，调用 `/x/v3/fav/resource/batch-del`。
- 请求成功后将条目标记为已移除；失败时保留条目并记录 B 站错误信息。

## 前端设计

复用预览页已有的“跳过条目”折叠面板：

- 新增“无访问权限”原因分组，支持独立折叠。
- 无标题条目优先显示 `BVID：...`；BVID 缺失时显示资源 ID，不显示空白标题。
- 无访问权限条目默认勾选为“可移除”。
- 点击移除后继续使用现有二次确认文案和成功/失败反馈。
- 空方案时仍显示预览页及跳过面板，不出现 AI 分类结果或错误的历史缓存条目。

## 兼容性与边界

- 历史会话不倒灌缓存；用户需要重新发起一次整理，才能补采集并列出此前未识别的无访问权限条目。
- 正常可读取的非视频资源仍按现有逻辑参与分类；本设计对其没有降级。
- 收藏夹元数据的 `media_count` 可能包含当前无法读取的资源，因此它代表“收藏关联数”，不等同于“可整理数”。

## 测试策略

1. B 站客户端：资源 ID 接口的返回规范化测试。
2. 会话采集：详情列表为空、ID 清单有两条时，生成两条可移除的 `inaccessible` 跳过记录，AI 不被调用，最终进入预览状态。
3. 会话采集：详情列表有一条、ID 清单有两条时，一条进入分类、一条进入跳过记录。
4. 缓存回退回归：会话资源为空但 `videos` 表存在同一 `fid` 的旧缓存时，AI 的输入仍为空，缓存不会进入方案。
5. 前端静态检查：无标题跳过条目优先显示 BVID，并保留跳过面板的勾选删除入口。
6. 完整测试套件及针对性会话测试。
