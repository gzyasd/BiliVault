# 最终方案执行进度设计

## 背景与根因

当前 `/api/session/{sid}/execute` 会同步等待整个移动过程结束后才返回。前端在此期间只把按钮改为“执行中...”。`ClassifySession.execute()` 虽然已有 `on_progress` 参数，但只在全部完成时发送一次 `done` 事件，因此无法展示中间进度。

## 目标

1. 最终执行方案时实时展示已处理数、总数、成功数、失败数和百分比。
2. 显示当前阶段、分类名称和来源收藏夹，用户能判断程序正在做什么。
3. 目标收藏夹创建阶段单独展示，不计入条目百分比。
4. 浏览器 SSE 短暂断开时，后台执行继续；重连后恢复最新进度。
5. 同一会话最多运行一个执行任务，避免重复移动。

## 后端设计

### 执行事件

`ClassifySession.execute()` 发送统一事件：

- `stage`: 固定为 `executing`，完成时为 `done`。
- `phase`: `creating_folders`、`moving` 或 `done`。
- `progress`: `processed / total`，范围 0 到 1。
- `processed`: 已经得到成功或失败终态的移动操作数。
- `total`: 本次实际需要执行的移动操作总数。
- `success`、`failed`: 当前累计结果。
- `category`、`source_fid`: 当前正在处理的分类和来源收藏夹；没有时为空。
- `folders_created`、`folders_total`: 创建目标收藏夹阶段的辅助进度。

同一资源若来自多个源收藏夹，会产生多次实际移动操作，因此 `total` 按 `move_groups` 中的操作数计算，不按资源组合键去重。

目标收藏夹创建失败时，该分类下对应的移动操作直接计为已处理且失败，并立即发送进度。每个移动批次成功或失败后发送一次事件。

### 执行任务与 SSE

新增 `/api/session/{sid}/execute/stream`：

- 当会话处于 `pending_review` 时启动后台执行任务。
- 当任务已运行时复用同一个任务，不重复执行。
- 用内存状态保存每个会话最新进度，SSE 客户端重连后先收到最新状态。
- SSE 断开只停止推送，不取消后台任务。
- 会话已经 `done` 时直接返回完成事件。
- 服务进程重启造成 `executing` 状态但没有后台任务时，返回明确错误，不尝试盲目重放不可逆操作。

保留原有 POST 执行接口以兼容旧调用，但新版前端使用 SSE 执行流。

## 前端设计

预览页底部操作区增加执行进度状态：

- 确认执行后隐藏普通确认按钮，显示进度条。
- 显示 `已处理 N / M`、成功数、失败数和百分比。
- 创建阶段显示“正在创建目标收藏夹（x/y）”。
- 移动阶段显示“正在处理：分类名 · 来源收藏夹 ID”。
- 收到 `done` 后关闭 SSE 并进入结果页。
- 收到 `fail` 后恢复操作区并显示错误。
- 打开状态为 `executing` 的可恢复会话时，直接重连执行进度流，不再次弹出确认框。

## 错误与安全

- 执行仍然只能从 `pending_review` 进入 `executing`。
- 断线重连不会创建第二个后台任务。
- 进度只在一个批次得到明确成功或失败结果后推进，不做前端估算。
- 不新增执行中取消功能，避免部分移动后产生难以解释的半完成状态。

## 测试

1. 会话执行测试：每个移动批次后发送真实 `processed/total/success/failed`。
2. 创建目标收藏夹失败测试：对应操作计入已处理和失败。
3. SSE 测试：连接启动后台任务、输出进度和完成事件，重复连接不启动第二个任务。
4. 前端静态测试：存在执行进度条、统计字段、SSE 地址及断线处理。
5. 运行完整 pytest 测试套件。
